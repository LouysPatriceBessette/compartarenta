## Why

The product currently treats "participants" as ad-hoc, temporary entries created inline inside each module (housing today, vehicle tomorrow). The same real person ends up duplicated across modules with copies of name and avatar, with no stable identity to reconnect them, and with no privacy-preserving way to verify "this is really the person my peer claims to be". A separate **Contacts** capability is needed as the **single source of truth for who the user knows**, so any module can reuse an existing contact instead of capturing a temporary name/avatar each time. Contacts also become the right place to define **how two users establish a connection** in the first place: out-of-band sharing of a unique code and a handshake performed via the encrypted relay defined in `privacy-first-sync-architecture`.

## What Changes

- Introduce a **Contacts module** as a distinct functional area separate from housing and vehicle, with its own list, editor, and detail view.
- Define a **Contact entity** independent from any module-specific "participant" concept. Each Contact has a stable on-device identifier, a display name, an avatar, and (for connected contacts) the public material and routing identifiers needed to send relay messages.
- Define **unique invitation codes** generated by a user's device, with a documented format suitable for being shared **outside the app** (text, email, QR — exact channel is user's choice).
- Define a **handshake protocol** using the existing encrypted relay (`relay-sync-no-persistence`) so that two users who exchange the invitation code (one direction is enough) can complete mutual contact establishment without the relay seeing plaintext identity.
- Define **module participation linkage**: each module references contacts (housing participants, vehicle participants, …) rather than embedding a temporary name/avatar. Local "unknown / temporary" placeholders are no longer used for **new** module entries created after this capability is in place.
- Define **migration** of existing module participants that already exist on-device into Contacts entries (auto-create a local-only Contact per unique participant, then re-point module references). Connected handshake remains an explicit follow-up action; migration alone does NOT auto-pair contacts across devices.
- Define **privacy and minimal disclosure rules** so a contact is identifiable to its owner without leaking unnecessary metadata to the relay or to peers.
- Define **deletion / unlink rules**: deleting a contact does not delete module data that already references the corresponding participant; the contact link becomes "archived/unknown" on the local module side but the module's accepted ledger remains intact (consistent with `data-locality-and-client-storage`).

## Capabilities

### New Capabilities

- `contacts-domain-model`: Local data model for Contacts (entity shape, connected vs local-only contact, relationship to modules' "participant" references).
- `contact-invitation-and-codes`: Generation, format, lifetime, and out-of-band sharing of unique invitation codes.
- `contact-handshake-over-relay`: Protocol-level requirements for the encrypted handshake that promotes a local-only contact to a connected contact, layered on `relay-sync-no-persistence`.
- `contacts-module-integration`: How housing, vehicle, and any future module reference contacts and stop authoring temporary inline participants.
- `contact-privacy-and-deletion`: Minimal disclosure of contact metadata, contact deletion/unlink rules, and impact on module ledgers.

## Impact

- Adds a new top-level area in the app shell ("Contacts").
- Introduces a new on-device table set (or equivalent storage) and migration from existing participant data.
- Establishes the **first** product use of the encrypted relay (`relay-sync-no-persistence`) end-to-end, before any housing-plan-related sync goes live.
- Becomes a precondition for the housing module to drop its inline temporary participant flow.
- Bound to `relay-server-infrastructure-and-audit` for the concrete server piece needed to deliver invitation/handshake messages.
